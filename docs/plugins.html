Plugins

Oxidian can run real Obsidian community plugins. This guide explains how the compatibility layer works, how to install plugins, and what to expect.

How Plugin Compatibility Works #

Oxidian includes a comprehensive Obsidian API compatibility shim ‚Äî a JavaScript module that reimplements the obsidian module API that community plugins import from. When a plugin does require('obsidian'), it gets Oxidian's implementation instead.

Architecture

  • Plugin Loader ‚Äî Discovers plugins in .obsidian/plugins/, reads their manifest.json and main.js
  • Sandboxed Execution ‚Äî Each plugin's main.js is executed in a sandboxed context with a simulated CommonJS module/exports/require environment
  • API Shim ‚Äî Implements the core Obsidian API classes: Plugin, App, Vault, Workspace, Notice, Modal, Setting, PluginSettingTab, Component, Events, TFile, TFolder, and more
  • Plugin Registry ‚Äî Tracks all registered commands, setting tabs, ribbon icons, post-processors, and status bar items
  • Command Palette Integration ‚Äî Plugin commands appear alongside built-in commands in the Ctrl+P palette

API Version

Oxidian's API shim targets Obsidian API version 1.7.2. Plugins targeting this version or earlier have the best compatibility.

Installing Plugins Manually #

Obsidian community plugins follow a standard structure. To install one manually:

Download the Plugin

Get the plugin files from the plugin's GitHub repository. You need at minimum: manifest.json and main.js. Many plugins also include styles.css.

Create the Plugin Folder

Inside your vault, create the directory .obsidian/plugins/<plugin-id>/. The plugin ID comes from the id field in manifest.json.

Copy the Files

Place manifest.json, main.js, and optionally styles.css into the plugin folder.

Enable the Plugin

Go to Settings ‚Üí Plugins. The plugin should appear in the list. Toggle it on to load it.

Using the Load Plugin Button #

Oxidian provides a convenient Load Plugin button in Settings ‚Üí Plugins:

  1. Click "Load Plugin"
  2. A folder picker dialog opens ‚Äî select the folder containing the plugin's manifest.json and main.js
  3. Oxidian reads the manifest, copies the plugin to your vault's .obsidian/plugins/ directory, and makes it available
  4. Enable the plugin in the list

If the file dialog is not available (e.g., on some minimal Linux setups), you can always install plugins manually as described above.

Stub Warnings

When a plugin calls an API method that isn't fully implemented, Oxidian logs a one-time warning to the browser console:

text 
[Oxidian] Stub: Workspace.getLeavesOfType not fully implemented

This helps you identify which features a plugin expects but doesn't have. Most plugins work fine with partial implementations.

Known Limitations #

  • Workspace Leaf API ‚Äî The full WorkspaceLeaf/WorkspaceSplit/WorkspaceTab model is not implemented. Plugins that manipulate pane layout directly may not work.
  • Editor API (CodeMirror) ‚Äî With HyperMark (v1.0+), Oxidian uses CodeMirror 6 for the focused block. Plugins using editorCallback and replaceSelection now work. However, plugins relying on CM6 decorations or view plugins across the entire document may have limited support since only the focused block uses CM6.
  • File suggest / Suggest modal ‚Äî Autocomplete modals are stubbed. Plugins relying on SuggestModal or FuzzySuggestModal have basic implementations.
  • Canvas / Publish / Sync APIs ‚Äî Not available (Oxidian doesn't have these features).
  • Mobile API ‚Äî No mobile-specific APIs.
  • External module requires ‚Äî Plugins that require() Node.js modules other than 'obsidian' will see an empty object and a console warning.
  • Styles ‚Äî Plugin styles.css is loaded, but CSS targeting Obsidian's specific class names may not match Oxidian's DOM structure.

Tip: If a plugin doesn't work, open the browser developer console (usually Ctrl+Shift+I ‚Üí Console tab in Tauri) to check for API stub warnings and errors. This tells you exactly which APIs are missing.
